## 프로젝트 개요

**프로젝트명**: pyRoad: 초등학생을 위한 인터랙티브 파이썬 학습 플랫폼

**핵심 목표**:
pyRoad는 초등학교 고학년(4~6학년, 10~12세) 학생을 대상으로 파이썬 프로그래밍을 재미있고 체계적으로 학습할 수 있는 웹 기반 교육 플랫폼이다.
AI 동물 마스코트 튜터("파이뱀 선생님")가 대화형으로 개념을 설명하고, 퀘스트 기반 게이미피케이션으로 학습 동기를 유지하며, 브라우저 내 Python 실행 환경(Pyodide)을 통해 별도 설치 없이 즉시 코딩을 시작할 수 있다.
커리큘럼은 기본 문법(print, 변수)부터 최종 셀프 프로젝트 만들기까지 단계적으로 구성되며, 마지막에는 LLM이 단계별로 안내하는 핸즈온 프로젝트를 수행한다.

**해결하려는 문제**:

| 문제 | 현재 상황 | 결과 |
|---|---|---|
| 성인 중심의 파이썬 교육 | 기존 온라인 강좌가 성인/대학생 대상으로 설계됨 | 초등학생은 용어, 속도, 난이도에서 이탈 |
| 한국어 초등학생 맞춤 플랫폼 부재 | 한국어로 된 아동 친화적 파이썬 학습 환경이 없음 | 영어 플랫폼에 의존하거나 학원에 의존 |
| 흥미 유지 어려움 | 텍스트 위주 강의, 단조로운 문제 풀이 | 초반 흥미 후 금세 지루해하며 학습 포기 |
| 환경 설정 장벽 | Python 설치, IDE 설정 등 사전 준비 필요 | 시작 전에 이미 좌절, 학부모 도움 필요 |

**핵심 가치 제안**:
- **재미**: 동물 마스코트 AI 튜터가 대화하며 가르치고, 퀘스트/보상 시스템으로 게임처럼 학습
- **쉬움**: 브라우저만 열면 바로 코딩 시작 (설치 불필요, Pyodide 기반)
- **체계**: 기본 문법부터 셀프 프로젝트까지 7단계 커리큘럼으로 단계적 성장
- **맞춤**: LLM이 학생의 수준과 상황에 맞는 설명, 힌트, 피드백을 동적으로 생성


**사용자 시나리오**:

1. 학생이 pyRoad 랜딩 페이지에 접속하면, 파이뱀 선생님 캐릭터가 환영 애니메이션과 함께 인사
2. Google 계정으로 로그인 (Supabase Auth)
3. 첫 방문 학생은 "학습 월드맵" 화면으로 이동 — 스테이지들이 잠금/해제 상태로 표시된 모험 지도
4. 학생이 해제된 스테이지(예: "변수의 숲") 클릭 → 해당 스테이지의 첫 퀘스트 시작
5. 퀘스트 화면: 파이뱀 선생님이 스토리와 함께 새 개념을 대화형으로 설명 (LLM이 커리큘럼 뼈대를 기반으로 동적 생성)
6. 학생이 코드 에디터에서 직접 Python 코드를 작성하고 "실행" 버튼 클릭 → Pyodide가 브라우저 내에서 실행하여 결과 표시
7. 막히면 "힌트" 버튼 클릭 → 파이뱀 선생님이 단계적 힌트 제공 (1단계: 개념 넛지, 2단계: 구체적 방향, 3단계: 거의 정답 수준)
8. 퀘스트 완료 시: XP 획득, 레벨업 가능, 뱃지 조건 충족 시 뱃지 획득, 다음 퀘스트/스테이지 잠금 해제
9. 진행 상황은 Supabase에 자동 저장 — 언제든 이어서 학습 가능
10. 마지막 스테이지("프로젝트 왕국"): LLM이 단계별로 안내하며 학생이 처음부터 하나의 미니 프로젝트를 완성 (예: "숫자 맞추기 게임을 만들어보자 → 1단계: random 모듈 사용법 → 2단계: 사용자 입력 받기 → ...")


## 커리큘럼 구조

콘텐츠 방식은 **하이브리드**로 운영한다:
- **사전 정의**: 커리큘럼 뼈대(스테이지, 퀘스트 순서, 학습 주제, 핵심 개념, 코드 검증 기준)는 개발자가 미리 작성하여 Supabase DB에 저장
- **동적 생성**: LLM이 커리큘럼 뼈대를 기반으로 스토리텔링, 개념 설명, 힌트, 피드백을 학생에게 맞춰 동적 생성

### 스테이지 로드맵

| 스테이지 | 테마명 | 학습 목표 | 퀘스트 수 |
|:---:|---|---|:---:|
| 1 | 파이썬 마을 입구 | `print()`, 주석(`#`), 기본 입출력(`input()`) | 3~4 |
| 2 | 변수의 숲 | 변수 선언, 자료형(`int`, `str`, `float`), 형변환 | 3~4 |
| 3 | 조건의 갈림길 | `if`/`elif`/`else`, 비교연산자, 논리연산자 | 3~4 |
| 4 | 반복의 동굴 | `for`, `while`, `range()`, `break`/`continue` | 3~4 |
| 5 | 리스트 호수 | 리스트 생성, 인덱싱, 슬라이싱, 기본 메서드(`append`, `len`) | 3~4 |
| 6 | 함수의 탑 | `def`, 매개변수, `return`, 기본 내장함수 | 3~4 |
| 7 | 프로젝트 왕국 | 종합 프로젝트 — LLM 가이드 빌딩 (멀티스텝) | 1 |

> **참고:** 각 퀘스트는 (1) 스토리 도입, (2) 개념 설명, (3) 코딩 실습, (4) 검증/피드백으로 구성된다.

### 퀘스트 구성 원칙
- 각 퀘스트의 **뼈대 데이터**(prompt_skeleton)에는 학습 주제, 목표, 실습 설명, 기대 출력값이 포함됨
- LLM은 이 뼈대를 기반으로 캐릭터 톤에 맞는 스토리와 설명을 동적 생성
- 스테이지 잠금 해제는 순차적: 스테이지 N의 모든 퀘스트를 완료해야 스테이지 N+1 해제
- 최종 프로젝트는 자유 선택이 아닌 **가이드 빌딩** 방식: LLM이 단계별로 안내하며 학생이 구현


## AI 튜터 페르소나 및 프롬프트 설계

### 캐릭터 설정

| 항목 | 내용 |
|---|---|
| 이름 | 파이뱀 선생님 |
| 종류 | 친근한 뱀 캐릭터 (파이썬 로고의 뱀 모티브) |
| 성격 | 친근하고 인내심 있으며 유머러스함. 실수해도 절대 혼내지 않고 격려 |
| 말투 | 친근한 존댓말 ("안녕! 오늘은 변수에 대해 배워볼 거예요~"), 짧은 문장, 이모지 적절 사용 |
| 핵심 원칙 | **절대 정답을 직접 알려주지 않음** — 항상 힌트와 유도 질문으로 학생이 스스로 답을 찾도록 안내 |

### 프롬프트 유형

| 프롬프트 유형 | 용도 | 호출 시점 |
|---|---|---|
| `quest_intro` | 퀘스트 도입 스토리 및 개념 설명 생성 | 퀘스트 시작 시 |
| `hint_generator` | 단계적 힌트 생성 (3단계 에스컬레이션) | 학생이 힌트 요청 시 |
| `code_feedback` | 코드 실행 결과에 대한 피드백 | 코드 제출 후 |
| `encouragement` | 성공/실패 시 격려 메시지 | 퀘스트 완료/실패 시 |
| `project_guide` | 최종 프로젝트 단계별 가이드 | 프로젝트 왕국 스테이지 |

### 힌트 에스컬레이션 전략

| 힌트 단계 | 수준 | 예시 (변수 할당 문제) |
|---|---|---|
| 1단계 | 개념 넛지 | "변수는 상자에 이름표를 붙이는 거예요. 어떤 이름표를 붙일까요?" |
| 2단계 | 구체적 방향 | "파이썬에서 이름표를 붙일 때는 `=` 기호를 사용해요. `이름 = 값` 형태로요!" |
| 3단계 | 거의 정답 | "`name = ____` 여기서 빈칸에 자기 이름을 넣어보세요!" |

### 안전 가드레일
- 부적절한 콘텐츠 생성 차단: 시스템 프롬프트에 아동 안전 지침 포함
- 응답 길이 제한: 최대 300토큰 (아동의 집중력 고려)
- LLM 실패 시 폴백: 퀘스트 뼈대 데이터에 사전 작성된 기본 설명을 표시
- 학습 범위 외 질문 대응: "지금은 파이썬 공부에 집중하자!" 류의 부드러운 리디렉션

### 프롬프트 구성 원칙
- 각 LLM 호출은 (1) 시스템 프롬프트(페르소나 + 안전 지침), (2) 커리큘럼 컨텍스트(현재 스테이지/퀘스트/개념), (3) 학생 코드 및 에러(해당 시), (4) 대화 히스토리(최근 N턴)으로 구성
- 프롬프트 템플릿은 `configs/prompts/` 디렉토리에서 관리


## 게이미피케이션 시스템

### XP(경험치) 시스템
- 퀘스트 완료 시 기본 XP 획득 (퀘스트별 `xp_reward` 값)
- 보너스 XP: 힌트를 적게 사용할수록 추가 XP (예: 힌트 0회 → +50%, 1회 → +25%, 2회 → +0%)
- XP는 누적되어 레벨업에 사용

### 레벨 시스템

| 레벨 | 이름 | 필요 누적 XP |
|:---:|---|:---:|
| 1 | 코딩 새싹 | 0 |
| 2 | 코딩 탐험가 | 100 |
| 3 | 코딩 모험가 | 300 |
| 4 | 코딩 용사 | 600 |
| 5 | 코딩 마법사 | 1000 |

> **참고:** 레벨 수와 필요 XP는 초기 기본값이며, 커리큘럼 운영 후 조정 가능하도록 설계한다.

### 뱃지 시스템

| 뱃지명 | 획득 조건 | 아이콘 테마 |
|---|---|---|
| 첫 코드 | 첫 번째 퀘스트 완료 | 발자국 |
| 힌트 마스터 | 힌트 0회로 퀘스트 완료 | 별 |
| 스테이지 클리어 | 한 스테이지의 모든 퀘스트 완료 | 깃발 |
| 프로젝트 빌더 | 최종 프로젝트 완료 | 왕관 |
| 연속 학습 3일 | 3일 연속 퀘스트 완료 | 불꽃 |

### 진행도 추적
- 월드맵에서 완료/진행중/잠금 상태를 시각적으로 표시
- 프로필 페이지: 총 XP, 현재 레벨, 획득 뱃지, 완료 스테이지 요약

> **의도적 제외:** 경쟁적 리더보드는 MVP에서 제외 (아동 심리적 부담 방지)


## 코드 실행 환경

### Pyodide 통합 구조
- **실행 방식**: Pyodide(WebAssembly 기반 CPython)를 Web Worker에서 실행하여 메인 UI 스레드 비차단
- **에디터**: CodeMirror 6 기반 코드 에디터 (Python 구문 하이라이팅, 아동 친화적 간소화 툴바)
- **출력 패널**: stdout/stderr 캡처하여 친근한 UI로 표시

### 보안 및 성능 제약
- 실행 타임아웃: **5초** (무한루프 방지, 초과 시 "코드가 너무 오래 걸려요!" 안내)
- 허용 모듈: 기본 내장 모듈만 (`math`, `random`, `string` 등)
- 차단 모듈: 파일 I/O(`open`, `os`), 네트워크(`urllib`, `socket`) 등
- Pyodide WASM 번들 (~10MB): 학습 페이지 진입 시 **Lazy Loading** + 로딩 중 캐릭터 애니메이션 표시

### 코드 검증 방식
- 각 퀘스트의 `validation_type`에 따라 검증:
  - `output_match`: Pyodide 실행 결과를 `expected_output`과 비교
  - `contains`: 출력에 특정 문자열 포함 여부 확인
  - `code_check`: 코드에 특정 패턴 포함 여부 확인 (예: `for` 문 사용했는지)
- 검증 실패 시: LLM이 `code_feedback` 프롬프트로 구체적 피드백 생성
- 부분 정답 인정: 올바른 접근이면 LLM이 격려하며 남은 부분 안내

## 데이터 모델 가이드 (Supabase)

### 테이블 구성

**`users`** — 사용자 프로필 (Supabase Auth와 연동)

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | uuid (PK) | Supabase Auth UID |
| `display_name` | text | 표시 이름 |
| `avatar_url` | text | 프로필 이미지 URL |
| `total_xp` | integer | 누적 경험치 |
| `current_level` | integer | 현재 레벨 |
| `created_at` | timestamptz | 가입일 |
| `last_login_at` | timestamptz | 마지막 접속일 |

**`stages`** — 커리큘럼 스테이지 정의 (개발자가 사전 입력)

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | uuid (PK) | 스테이지 ID |
| `order` | integer | 순서 |
| `title` | text | 스테이지명 (예: "변수의 숲") |
| `theme_name` | text | 테마 식별자 |
| `description` | text | 스테이지 설명 |
| `is_final` | boolean | 최종 프로젝트 스테이지 여부 |

**`quests`** — 퀘스트 정의 (개발자가 사전 입력, 커리큘럼 뼈대)

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | uuid (PK) | 퀘스트 ID |
| `stage_id` | uuid (FK) | 소속 스테이지 |
| `order` | integer | 스테이지 내 순서 |
| `title` | text | 퀘스트명 |
| `concept` | text | 학습 개념 키워드 |
| `prompt_skeleton` | jsonb | LLM에 전달할 뼈대 데이터 (주제, 목표, 실습 설명, 검증 기준 등) |
| `validation_type` | text | 검증 방식 (`output_match`, `contains`, `code_check`) |
| `expected_output` | text | 기대 출력값 또는 검증 패턴 |
| `xp_reward` | integer | 완료 시 기본 XP |

**`user_progress`** — 학생별 퀘스트 진행 상태

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | uuid (PK) | |
| `user_id` | uuid (FK) | 사용자 |
| `quest_id` | uuid (FK) | 퀘스트 |
| `status` | text | `not_started` / `in_progress` / `completed` |
| `hints_used` | integer | 사용한 힌트 수 |
| `code_submitted` | text | 최종 제출 코드 |
| `completed_at` | timestamptz | 완료 시간 |

**`user_badges`** — 획득한 뱃지

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | uuid (PK) | |
| `user_id` | uuid (FK) | 사용자 |
| `badge_type` | text | 뱃지 종류 (`first_code`, `hint_master`, `stage_clear`, `project_builder`, `streak_3`) |
| `earned_at` | timestamptz | 획득 시간 |

### RLS(Row Level Security) 정책
- `users`: 본인 데이터만 읽기/수정 가능
- `stages`, `quests`: 모든 인증 사용자가 읽기 가능 (쓰기 불가, 개발자가 직접 시딩)
- `user_progress`, `user_badges`: 본인 데이터만 읽기/쓰기 가능


## 아동 친화적 UX 가이드라인

### 색상 및 테마
- 밝고 따뜻한 색상 팔레트 (자연/모험 테마 — 숲, 호수, 동굴 등 커리큘럼 테마와 일치)
- 네온 컬러 지양, 눈에 편안한 파스텔 + 포인트 컬러 조합
- 다크 모드는 MVP에서 제외

### 타이포그래피
- 본문 최소 16px, 제목은 더 크게
- 둥글고 친근한 한국어 서체 (예: Pretendard 또는 Noto Sans KR)
- 코드 에디터: 가독성 좋은 모노스페이스 서체

### 인터랙션
- 모든 클릭 타겟 최소 44px (터치 디바이스 대응)
- 모든 액션에 명확한 시각적 피드백 (버튼 클릭, 코드 실행, 정답 등)
- 성공 시: 축하 애니메이션 (confetti, 캐릭터 춤, XP 획득 이펙트)
- 실패 시: 부드러운 격려 (절대 빨간 경고가 아닌, 파이뱀 선생님의 응원 메시지)

### 언어
- 쉬운 한국어, 짧은 문장
- UI 레이블에 기술 전문 용어 지양 (예: "실행" 대신 "코드 돌려보기!", "제출" 대신 "완료!")
- 에러 메시지도 파이뱀 선생님이 안내하는 형태

### 로딩 및 대기
- Pyodide 초기 로딩(~10MB): 파이뱀 선생님이 준비하는 애니메이션 + 재미있는 코딩 팁 표시
- LLM 응답 대기: 파이뱀 선생님이 생각하는 애니메이션

### 반응형 디자인
- 모바일/태블릿 대응 필수 (초등학생은 태블릿 사용 비율 높음)
- 퀘스트 페이지의 split view는 모바일에서 탭 전환 방식으로 변환
- 최소 지원 해상도: 360px 너비
